# Concept Document Template

This is the standard structure for the final output of a desire-to-design session.

## Principles

- **One page if possible**, two pages max
- **Skimmable** — clear headers, tables for decisions, no walls of text
- **Action-oriented** — ends with concrete next steps
- **Suitable for sharing** — can be used to test concept appeal with friends, communities, or potential users
- **Captures decisions, not aspirations** — don't list "things to consider," list "things decided"

## Standard sections

```markdown
# [Project Name] - Concept Document v0.1

> One-line tagline that makes someone want to know more

---

## Core concept

### Setting/Context
2-3 sentences establishing the world, market context, or content territory.

### Main subject (character/user/protagonist/customer)
The central figure or user. Be specific about who, not just what.

> Pull out the "gap" or unique angle here. What makes this subject interesting?

### Visual/aesthetic identity
What does it look/feel/sound like? Reference real works for shorthand.

---

## The core experience

### One-line description of the loop
What does the user/player/reader do, and why is it compelling?

### Visual flow
```
[Stage 1] → [Stage 2] → [Stage 3] → [Stage 4]
```

A simple diagram of the core loop.

### Key mechanic/feature/element
The ONE thing this project does that nothing else does quite the same way.

---

For each major decision, capture what was chosen and why:

| Decision | Choice | Why |

| | | |
| | | |

### Market position
Where this sits in the existing landscape. What's the gap it fills?

### Key differentiators

2. **Second differentiator**
3. **Third differentiator**

---

## Target audience

### Primary
Who will love this most? Be specific about existing communities.
```

## Example: filled-out version

See the game concept document produced in the original session for a complete example. Key qualities:

- Tagline made the project sound like something specific, not generic
- Each table row captured a real decision, not options under consideration
- Reference works were named (Slay the Spire, 우마무스메, etc.) — concrete, not abstract
- Next steps were small and immediate (start concept art, test on social media)
- Personal motivation (e.g., a partner as inspiration) was acknowledged in private notes, not the public doc

## What NOT to include

- Long lists of "potential features"
- Vague aspirations ("the game will be amazing")
- Marketing language ("revolutionary new approach")
- Justifications for decisions that don't add information
- Multiple competing visions ("could be A or B or C")
- Anything that wouldn't survive the question "is this decided or just considered?"
