# Example Session Flow

This is an annotated example of the desire-to-design skill in action, based on a real session that successfully shaped a vague desire into a concrete game concept.

## Initial state

**User**: 8-year backend developer, Korean, expressed multiple disconnected desires:
- "I want to start an agent-based company"
- "What if we make a problem-finding business?"
- "Maybe a puzzle game?"

This pattern (multiple unrelated domains) is the strongest signal that the user needs **Stage 1: Honor + redirect** before anything else. They don't yet know what they want.

## Stage 1: Honor and redirect

**Skill move**: Did NOT immediately help with any of the three ideas. Instead:
- Acknowledged the desire as common
- Honestly assessed the difficulty of each
- Identified the underlying motivation (escape current job? build wealth? make something they love?)

**Key insight surfaced**: The user's three ideas had nothing in common except "not my current job" — meaning the real question wasn't "which of these?" but "what do I actually want?"

**Skill move**: Asked the motivation-clarifying question:
- More money → optimize current career
- Freedom → freelancing/remote
- Make something → side project
- Big wealth → startup with risk tolerance

This reframe was critical. Without it, we'd have spent the session making a bad business plan.

## Stage 2: Hidden assets

**User said**: "I've invested a lot in playing games."

Most assistants would treat this as background. The skill **reframed it as their primary asset**:
- Game taste detection is rare
- It's the #1 thing most game developers struggle with
- This is a competitive edge in game development

This reframe transformed the user's self-perception from "wannabe game maker" to "person with calibrated taste who could make games."

## Stage 3: Taste analysis (longest stage)

**User mentioned Stardew Valley first.** Critically, when asked why, they said:
> "Not because it's a farming game — because of pixel art lightness AND the feeling of growth."

This is gold. The user **automatically separated surface (farming) from essence (lightness + growth)**. That's a sign of strong taste calibration.

**Skill move**: Reflected this back and dug deeper:
- "What kind of growth?" Numbers, unlocks, optimization, completion?
- User said "all of them complexly"
- Skill pushed: "Pick the dominant one when you said 'one more hour.'"

**User added Slay the Spire**: "Deck-building dopamine."
**User added the killer probe themselves**: "But there were similar games I didn't enjoy. What's the one-bit difference?"

That last question — coming from the user — was the breakthrough moment. The skill answered with a framework:
- Feedback immediacy (juice)
- Decision weight (every choice should be interesting hard)
- Pacing (when does session naturally end?)
- Next-run hook (after death, why come back?)
- Visual/audio identity

This gave the user a **vocabulary for analyzing games**, which is what they actually needed.

**Fun DNA extracted**: "The pleasure of stuff I built becoming visibly stronger" — combining StS deck synergy with SDV wealth accumulation. This is the rogue-lite + meta progression genre.

## Stage 4: Concept generation from taste

**User finally proposed concrete ideas**:
1. Auto-farming with monster employees
2. Boxer girl raising sim with deck-building
3. Office survival with deck-building

**Skill move**: Honest assessment, not validation:
- #1: Viable but content-explosive (auto games need huge content)
- #2: BEST option — empty market space, clear loop, fits taste DNA
- #3: Weakest — no clear climax, similar to #2 but worse

Recommended #2 with explicit reasoning. User accepted.

## Stage 5: One-line distillation

Iterated to: **"Raise a rookie boxer girl in a magic world to fight strange opponents with cards."**

This is specific, evocative, and unique. Could fit on a Steam page tagline.

## Stage 6: System decisions with tradeoffs

Made many decisions. Two examples of where tradeoff awareness mattered:

**User proposed**: "Let player choose deck size freely."

**Skill move**: Pushed back honestly. "Unlimited deck size collapses to one optimum (smallest deck, highest draw probability). You think you're giving freedom but you're killing variety." Suggested: 30 cards fixed, with later unlock to ±5. User accepted with understanding of the why.

**User proposed**: "Skip stamina for enemies, just player has it."

**Skill move**: Strong validation. "This is a smart 1-dev decision. Adding enemy stamina multiplies complexity exponentially without adding much. Future expansion possibility, not v1." Explicit positive reinforcement of good design instincts.

## Stage 7: Reality check

When user mentioned the protagonist would be modeled after their wife, the skill:
- Honored this as powerful motivation fuel
- BUT warned about the trap of too-personal-to-ship
- Suggested: keep wife as private inspiration, design character to be more universally appealing
- Reference: Eric Barone's wife inspired Stardew Valley but isn't directly in it

This let the user keep their meaningful motivation while making a marketable product.

## Final output

Compiled a one-page concept document with:
- Tagline
- Decision tables (not "considering X")
- Concept art direction
- Realistic 18-24 month roadmap
- Three immediate next actions

User explicitly stated: "I want to use this for concept art and to test market reaction." The skill optimized the document for this stated purpose, not for theoretical "best practices."

## What went RIGHT in this session

1. **Resisted the urge to help with the first idea proposed**. Instead spent the first 30% on motivation and self-discovery.
2. **Validated the user's accidental insight** (StD vs farming distinction) and built on it rather than steering past.
3. **Used real reference works heavily** (Stardew Valley, Slay the Spire, Brotato, Hades, Balatro) instead of abstract principles.
4. **Pushed back honestly on weak ideas** without being dismissive.
5. **Honored the personal motivation** without letting it derail the commercial viability.
6. **Ended with concrete, small next steps** ("download Godot, do tutorial"), not just the doc.

## What to watch for

- User wanted to skip ahead at multiple points; the skill held the structure when needed but flexed when the user was ready
- User asked "is my idea a genius idea?" — skill answered the meta-question (no genius required, finishing matters more) instead of literal answer
- User shared something personal (wife as inspiration); skill held this carefully, neither dismissed nor centered it inappropriately

## Pattern this session illustrates

This was a **pure desire→design** session. User had energy but no direction. By the end:
- Clear project (genre, mechanics, aesthetic)
- Connected to their taste (so they'll love working on it)
- Realistic scope (1.5-2 years solo)
- Concrete next action (concept art + Godot tutorial)
- Personal meaning preserved (wife as private fuel)

The output isn't just a document — it's a *project they're equipped to actually start*.
